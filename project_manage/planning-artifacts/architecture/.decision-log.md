### Décisions de structure prises pendant la rédaction

#### D1 — Posture « document de consolidation » plutôt que « document de découverte »

**Quoi** : `architecture.md` est rédigé comme une **synthèse** des choix déjà figés, pas comme une nouvelle discovery. Le document pointe vers les 3 docs source pour les détails plutôt que de les répéter.
**Pourquoi** : Les 3 docs d'archi étaient déjà rédigées en profondeur avant le pipeline BMAD (le projet a démarré côté doc avant d'installer BMAD). Re-découvrir les mêmes choix en 8 steps interactifs n'apporte aucune valeur — Winston (skill `bmad-create-architecture`) ne ferait que reformuler.
**Risque** : Si une décision a évolué entre les 3 docs et la réalité, la consolidation peut masquer la divergence.
**Mitigation** : `architecture.md § 16.1` confirme explicitement que les 3 docs source sont l'état de référence ; toute divergence future remontera en revue.
**Impact aval** : Les skills aval (`bmad-create-epics-and-stories`, `bmad-dev-story`) consultent en priorité `architecture.md` + ADRs, puis descendent dans les docs source au besoin.

#### D2 — Mapping explicite features mobile ↔ capabilities SPEC ↔ phases MVP

**Quoi** : `architecture.md § 5.2` aligne les 9 features mobile (`auth`, `content`, `exercises`, `billing`, `academic_health`, `gamification`, `chat`, `notifications`, `sharing`) sur les 6 CAP du SPEC et les 6 phases MVP.
**Pourquoi** : Sans ce mapping, John (PM en epics) pourrait créer des epics qui ne correspondent pas aux features architecturales. L'alignement préventif évite ce désalignement.
**Impact aval** : Chaque epic produit par `bmad-create-epics-and-stories` se mappe sur 1 feature mobile (ou 1 transversale).

#### D3 — 10 ADRs formalisés couvrant les décisions structurantes

**Quoi** : Création de 10 fichiers ADR dans `adrs/`. Chacun suit le format Contexte / Décision / Conséquences / Détail / Décisions liées.
**Pourquoi** : Les choix critiques (Riverpod vs GetX, Firebase complet, idempotence, etc.) méritent un format dédié et indexable. Une ADR fournit le **pourquoi** durable qui ne se trouve pas dans la doc d'archi (qui décrit le « comment »).
**Sélection** :

1. Flutter + Clean Architecture (le socle)
2. Riverpod vs GetX (arbitrage majeur)
3. Firebase complet (lock-in assumé)
4. BMAD method (méta-décision pilotage)
5. `doc/partage/` (frontière inter-équipes)
6. Sous-système figé inscription (décision produit structurante)
7. Mobile Money via agrégateur (décision externe critique)
8. Idempotence via sessionId (pierre angulaire technique)
9. `flutter_smooth_markdown` wrappé (mitigation risque)
10. Pas de cache custom (engagement Firestore natif)

**Écartés intentionnellement** :

- Choix Lucide pour les icônes — déjà dans DESIGN.md, pas une décision d'archi.
- Compression WebP pour photos Mode 1 — déjà dans NFR-4 du PRD, pas une décision d'archi.
- Choix de `dio` vs `http` — déjà tranché dans Mobile Package Architecture § 3, mineur.

#### D4 — `architecture.md` au lieu de `solution-architecture.md` ou autre nom

**Quoi** : Nom de fichier `architecture.md` dans `architecture/`.
**Pourquoi** : Convention BMAD (template `architecture-decision-template.md` produit `architecture.md`). Sub-folder `architecture/` pour cohérence avec les autres skills (chacune dans son sous-dossier de `planning-artifacts/`).
**Impact aval** : Les skills aval cherchent `architecture.md` par convention.

#### D5 — ADRs dans sous-dossier `adrs/` plutôt que `architecture/`

**Quoi** : `adrs/ADR-NNN-titre-kebab.md` dans un sous-dossier dédié.
**Pourquoi** : Convention industrielle (Anti-corruption layer / ThoughtWorks / Spotify). Permet aussi à un éditeur Markdown de présenter les ADRs en index séparé.
**Impact aval** : Bob/Amelia/Winston en future référence à un ADR utilisent `[ADR-008](adrs/ADR-008-idempotency-via-sessionid.md)`.

#### D6 — `architecture.md` § 13 « Risques techniques identifiés » avec 10 risques cotés

**Quoi** : Section dédiée aux risques techniques avec 4 élevés, 4 moyens, 2 surveillés. Chacun avec mitigation.
**Pourquoi** : Le PRD couvre les Open Questions et Assumptions de **produit**. L'architecture documente les risques de **mise en œuvre** : agrégateur, package jeune, latence, conformité curriculum, etc.
**Impact aval** : Winston peut référencer ces risques en cas de découverte d'un problème en P1/P2/P3.

#### D7 — `architecture.md` § 15 « Mapping NFRs PRD → exigences techniques »

**Quoi** : Tableau qui pour chaque NFR du PRD (NFR-1 à NFR-15) liste l'exigence technique et le composant qui l'applique.
**Pourquoi** : Sans ce mapping, un dev qui implémente une story pourrait oublier un NFR. Le mapping est l'outil de vérification que toute exigence non-fonctionnelle a un emplacement.
**Impact aval** : Dans `bmad-check-implementation-readiness`, ce tableau sert de checklist.

#### D8 — Pas d'agrégateur Mobile Money pré-choisi dans l'archi

**Quoi** : `architecture.md` mentionne « Tranzak / Campay / MyCoolPay » comme candidats, mais ne tranche pas. Reporté à la décision PRD OQ-10.
**Pourquoi** : Le choix dépend de critères opérationnels (frais, webhook signature, support) qui sortent de l'architecture pure. C'est une décision **business** + **compliance** + **technique**, à prendre par l'équipe (PM + lead technique) sur la base de tests de bout en bout en sandbox.
**Risque** : Bloque la P4 si non résolu en P3.
**Mitigation** : Démarche d'ouverture compte marchand à lancer dès J1 (cf. risque R1 et ADR-007).

#### D9 — Pas d'architecture admin / landing dans ce document

**Quoi** : `architecture.md` § 4.3 mentionne « à décider » pour stack admin et landing. Pas de spécification ici.
**Pourquoi** : Ce sont d'autres dépôts avec leurs propres équipes et décisions. Les contraindre depuis ce document mobile serait inapproprié.
**Impact aval** : L'équipe admin/landing créera ses propres `architecture.md` quand elle démarrera.

#### D10 — Aligner les types Dart ↔ TS par discipline manuelle + tests de contrat (pas de génération auto)

**Quoi** : Stratégie 1+3 de l'archi backend § 8.2 retenue. Pas de codegen depuis un schéma central (OpenAPI / JSON Schema).
**Pourquoi** : ~12 contrats au MVP, le coût d'une génération auto est supérieur au bénéfice. Si le périmètre dépasse ~30 contrats post-MVP, ouvrir un nouvel ADR.
**Risque R7** documenté en `architecture.md § 13.2`.

**Auteur** : DelRoos / Claude

**AC2 (rendu visuel)** : ❌ KO sur les 3 cours.

### Diagnostic préliminaire (à valider)

Trois hypothèses, classées par probabilité :

1. **Conflit de scroll** (probable) : `SmoothMarkdown` retourne un `Column(crossAxisAlignment: stretch)`. Dans le `TestCourseDetailPage`, on l'enveloppe d'un `ListView`. Le rendu serait correct seul mais sale dans ce nesting (contraintes verticales infinies + cross-axis stretch). À tester : remplacer `ListView` par `SingleChildScrollView` + `Column`, ou `CustomScrollView` + `SliverToBoxAdapter`.
2. **`MarkdownStyleSheet.fromTheme` non-fonctionnel sur notre ThemeData** (possible) : notre `buildLightTheme()` consomme `AppTypography` et `AppColors` qui peuvent ne pas matcher ce que `MarkdownStyleSheet.fromTheme` attend. À tester : passer `MarkdownStyleSheet.light()` directement (style par défaut du package) → si ça rend, le bug est dans le mapping de theme.
3. **R8/minification release strippe quelque chose** (peu probable, packagepur Dart) : à exclure en testant un build release **après** désactivation R8 (gradle `minifyEnabled false`).

### Décision AC4 — Provisoire : **continuer avec investigation prioritaire**

Le KO visuel ne suffit PAS à conclure "fallback vers assemblage classique" — les 3 hypothèses ci-dessus sont toutes des bugs **côté projet** (mauvaise intégration) plutôt que **côté package**. Si après les 3 tests d'investigation le package rend correctement, on continue. Sinon, on bascule.

**Action immédiate** (story de suivi, 0.19.1 ou directement 0.19 phase terrain bis) :
1. Tester `MarkdownStyleSheet.light()` direct (hypothèse 2) — 5 min
2. Tester `SingleChildScrollView` + `Column` (hypothèse 1) — 5 min
3. Tester un cours minimal (3 lignes de markdown) au cas où c'est la taille — 5 min
4. Si tout reste KO : passer aux exemples officiels du package (`example/` du repo upstream) pour isoler.

**Si après 1-2h de debug toutes les hypothèses tombent** → décision « fallback » officialisée : story de remplacement E2 avec `flutter_markdown` + `flutter_math_fork` + WebView Mermaid. Estimation +1-2j.

### Conclusion de l'investigation

- Le wrapper `PedagogicalContent` + `SmoothMarkdown` **fonctionne** sur markdown basique.
- Le wrapper **échoue silencieusement** sur nos 3 fichiers `.md` réels (LaTeX inline/display, blockquotes, tableaux, listes imbriquées, blocs de code, Mermaid).
- Aucune erreur Flutter visible en `logcat` release → le rendu retourne `SizedBox` ou un Container vide silencieusement.
- Les builders `inline_math` / `block_math` (via `flutter_math_fork`) sont enregistrés dans le default registry du package. Le rendu LaTeX devrait fonctionner.
- Le builder `mermaid` n'est PAS dans le default registry — l'exemple officiel le passe explicitement via `plugins: ParserPluginRegistry()..register(MermaidPlugin())` + `builderRegistry: BuilderRegistry()..register('mermaid', MermaidBuilder())`. Notre `PedagogicalContent` n'expose pas cette surface.

### Hypothèses restantes (non testées)

1. **flutter_math_fork lève une exception sur une formule particulière** (par ex. `\lim_{h \to 0}`, `\dfrac`, ou un de nos cas spécifiques). Si caught silencieusement par `SmoothMarkdown`, le widget entier pourrait dégrader en bloc vide. **À tester** : injecter un cours réduit avec UNE seule formule LaTeX simple (`$x^2$`), puis ajouter une formule à la fois.
2. **Le bloc Mermaid (sans plugin enregistré) casse tout le parse en aval** : si le parser bute sur `\`\`\`mermaid` sans plugin, il peut entrer dans un état corrompu qui n'émet plus rien. **À tester** : tronquer `info_algo_recherche.md` aux 200 premiers caractères (sans Mermaid) → si ça rend, c'est lui.
3. **Caractères non-ASCII en grand nombre** (français accentué, emoji 🟢) cassent un builder. **Peu probable** mais à exclure.
