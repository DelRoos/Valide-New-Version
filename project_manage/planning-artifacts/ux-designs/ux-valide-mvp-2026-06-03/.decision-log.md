### Décisions de distillation prises pendant la rédaction

#### D1 — Le Design System HTML est la source unique de DESIGN.md, lifté littéralement

**Quoi** : Tous les tokens visuels de `DESIGN.md` (couleurs hex, polices, échelles, composants visuels) sont **extraits directement** des `:root { --primary: #2563EB; ... }` et des classes CSS du Design System HTML existant. Aucun token n'a été inventé.
**Pourquoi** : L'utilisateur a pointé explicitement ce fichier comme entrée. Il représente le travail de design déjà fait et validé en amont. Sally (skill `bmad-ux` en v6.8.0) doit **capturer**, pas **inventer**.
**Impact aval** : Quand l'équipe mobile implémentera les tokens en Dart (dans `core/theme/tokens.dart`), ils auront une correspondance 1:1 avec le HTML existant. Aucune divergence possible.

#### D2 — Échelle typographique étendue de 4 → 9 niveaux

**Quoi** : Le Design System HTML montre un Hero h1 (46 px), un h2 de section (30 px), un sub-head (13 px uppercase), et un type-scale interne. J'ai expliciter 9 niveaux nommés (`display`, `h1`, `h2`, `h3`, `body-strong`, `body`, `meta`, `caption`, `eyebrow`) en couvrant les usages typiques d'une app mobile.
**Pourquoi** : 4 niveaux ne suffisent pas pour les besoins de l'app (titre d'écran, titre de section, corps long, métadonnée, badge, etc.). L'extension est cohérente avec les graisses (400/600/700/800/900) du HTML.
**Risque** : Si le Design System HTML est mis à jour pour ajouter des niveaux différents, il faudra réconcilier. À tracer comme `[ASSUMPTION]` à valider visuellement.
**Mitigation** : Les noms sont sémantiques (`body-strong` plutôt que `text-16-700`) — restent valides quelle que soit la taille exacte.

#### D3 — Échelle d'espacement étendue à 10 valeurs (4, 8, 12, 16, 20, 24, 32, 40, 48, 64)

**Quoi** : Le Design System HTML annonce « grille 4 px » mais ne liste pas explicitement l'échelle. J'ai déclaré une progression standard 4 → 64 px.
**Pourquoi** : Échelle cohérente avec la grille 4 px annoncée et avec les valeurs déjà utilisées dans le HTML (16 / 24 / 32 / 64 px observés).
**Impact aval** : Sera implémenté dans `core/theme/tokens.dart` avec `flutter_screenutil` (les valeurs en `.h`, `.w`, `.sp`, `.r`).

#### D4 — 7 niveaux de rayons nommés au lieu de 3-4

**Quoi** : Le Design System HTML utilise des rayons à plusieurs valeurs (6, 9, 11, 14, 16, 18, 999 px observés). J'ai nommé 7 niveaux (`xs`, `sm`, `md`, `lg`, `xl`, `2xl`, `pill`).
**Pourquoi** : Cohérent avec ce qu'on observe dans le HTML. Permet d'avoir une grammaire claire (« lg pour boutons, 2xl pour cartes »).
**Règle ajoutée** : « ne pas mélanger 4 rayons différents sur le même écran » — privilégier `lg` + `2xl` comme paire dominante.

#### D5 — Bottom tab bar à 4 onglets (Accueil / Matières / Activités / Profil)

**Quoi** : Architecture top-level à 4 onglets dans une bottom tab bar. Pas de drawer hamburger.
**Pourquoi** : Cohérent avec les maquettes des modules M1-M18 (le Design HTML montre cette structure). Les 4 onglets couvrent les surfaces que l'élève visitera **quotidiennement** ; les sub-surfaces (santé scolaire, classements, chat, paywall) sont atteintes par drill-down ou actions contextuelles.
**Alternative écartée** : 5 onglets (séparer Activités et Examens). Le Design HTML montre que les examens sont une sous-section d'Activités, ce qui maintient une charge cognitive moindre.
**Impact aval** : Sally (UX) confirme cette IA dans le routing `go_router`.

#### D6 — Étiquettes santé scolaire `solide / à renforcer / priorité` (et non `bon / moyen / faible`)

**Quoi** : Lexique positivement orienté pour les trois étiquettes de santé.
**Pourquoi** : Cohérent avec le principe « Motivation » du Design System HTML (« encourager sans culpabiliser »). « Priorité » suggère une action, « Faible » suggère un échec. Cohérent aussi avec le Decoupage MVP qui utilise déjà ces termes.
**Impact aval** : Couvre toutes les surfaces (santé scolaire, recommandations, notifications). Verrouillé par le glossaire SPEC.

#### D7 — Couleur seule n'est jamais un signal — règle stricte

**Quoi** : Tout état (étiquette santé, statut paiement, erreur réseau) est **toujours doublé** d'un texte explicite et optionnellement d'une icône.
**Pourquoi** : Accessibilité (élèves daltoniens, vision diminuée). Cohérent avec le principe « Accessibilité » du Design System HTML. Standard WCAG.
**Impact aval** : Aucun composant `badge` ou `chip` ne s'affiche sans un texte.

#### D8 — Tutoiement français par défaut, ton équivalent direct en anglais

**Quoi** : Microcopie en FR utilise le tutoiement (« Tu peux »), en EN utilise un ton direct informel.
**Pourquoi** : Cible ado/jeune adulte, contexte familier d'app de pratique. Le vouvoiement créerait une distance émotionnelle inadaptée.
**Impact aval** : Toutes les chaînes ARB FR/EN à rédiger dans ce registre.

#### D9 — Pas d'émoji dans l'interface (sauf célébrations mesurées)

**Quoi** : Bannissement des émojis décoratifs dans les écrans courants.
**Pourquoi** : Le Design System HTML mise sur la sobriété (« moderne, clair, motivant » sans surcharge). Trop d'émojis = ton infantilisant ou marketing. Les rares cas autorisés : récap de réussite avec mention, montée de niveau (et même là, mesurés : un trophée Lucide vaut mieux que 🎉🎉🎉).
**Impact aval** : Sally et l'équipe Content valident ce choix dans la rédaction des chaînes.

#### D10 — Pas de dark mode en V1 — `[ASSUMPTION]` à valider

**Quoi** : Light mode only en V1.
**Pourquoi** : Le Design System HTML ne propose pas de palette sombre. Hypothèse que la consultation se fait majoritairement en journée par les élèves (lecture sur cahier en parallèle, lumière naturelle ou ampoule). Ajout d'un dark mode est un effort non trivial sans bénéfice marché clair en V1.
**Risque** : Les élèves qui révisent tard le soir peuvent ressentir l'inconfort.
**Mitigation** : Mettre dark mode candidate V2 si retour utilisateur. Logger dans `[ASSUMPTION]`.
**Impact aval** : `core/theme/app_theme.dart` ne déclare qu'un ThemeData light.

#### D11 — Streaks visibles sur profil mais sans notification de rupture

**Quoi** : La métrique « streak » (jours consécutifs d'activité) est visible sur le profil mais n'est **pas** au cœur de la gamification, et **ne génère pas** de notification culpabilisante en cas de rupture.
**Pourquoi** : Anti-pattern Duolingo explicite. Aller dans cette direction démotive les élèves qui ratent un jour (souvent pour des raisons hors de leur contrôle — coupure courant, examen IRL).
**Impact aval** : `gamification` feature ne crédite pas de bonus « streak conservé » au-delà du raisonnable. Pas de push « ton streak va se briser ! ».
**Risque** : ce choix de design peut réduire l'engagement quotidien des élèves « game-driven ». À mesurer post-MVP.

#### D12 — Paywall plein écran **avant** chargement de l'exercice premium

**Quoi** : Pour un compte gratuit qui tape Mode 2 / Mode examen / Fiches / Chat ×20, l'app affiche le paywall **avant** de charger l'exercice ou la ressource.
**Pourquoi** : (1) Économie data — pas de lecture Firestore inutile pour un contenu non accessible. (2) Sécurité — moins d'opportunités de fuiter du contenu premium par modification de l'app (cf. NFR-9 et FR-23 du PRD).
**Impact aval** : Le routing `go_router` vérifie le statut premium **avant** de naviguer vers les écrans premium. Cohérent avec l'archi mobile § 17.4.

#### D13 — Skeleton plutôt que spinner pour les chargements > 500 ms perçus

**Quoi** : Sur les écrans qui chargent du contenu structuré (cours, liste de matières, dashboard), utiliser un skeleton qui mime la forme finale plutôt qu'un spinner abstrait.
**Pourquoi** : Réduit la perception de lenteur (étude Nielsen Norman Group). Donne au cerveau quelque chose à pré-parser. Cohérent avec NFR-2 (démarrage < 3 s) — l'écran semble « presque prêt » même quand les données ne sont pas là.
**Impact aval** : Sally et Amelia (dev) implémentent un widget `Skeleton` réutilisable dans `core/widgets/`.

#### D14 — Pas de carrousel d'écrans top-level

**Quoi** : La bottom tab bar n'autorise pas le swipe horizontal entre les 4 onglets.
**Pourquoi** : Anti-pattern documenté — les utilisateurs « tombent » accidentellement sur un autre onglet en scrollant verticalement. La navigation par tap est plus prévisible.
**Impact aval** : Implémenter avec `PageView` désactivé pour le swipe ou `IndexedStack` qui n'a pas ce comportement.

#### D15 — Aucun mock HTML rendu en V1 — les spines suffisent

**Quoi** : Pas de rendu de key-screen mocks dans `.working/` ou `mockups/` pour cette première passe. Les spines (DESIGN + EXPERIENCE) suffisent.
**Pourquoi** : Le Design HTML existant (`Valide - Design.html`) sert déjà de référence visuelle pour les 18 modules. Ré-générer des mocks HTML serait redondant.
**Impact aval** : Sally pourra générer des mocks ciblés (en Update mode) si une feature spécifique le justifie en aval (ex. paywall qui mériterait une exploration A/B). Pour le MVP, les spines + le Design HTML existant suffisent.

#### D16 — Quatre breakpoints téléphone, aucun support tablette

**Quoi** : Mobile-only, avec breakpoints à 360 / 393 / 412 / 480 dp.
**Pourquoi** : Cohérent avec le Constraint « Android-first » du SPEC et le NFR-1 du PRD. Tablette = scope V2/V3.
**Impact aval** : Aucun layout responsif desktop/tablette en V1. Si lancé sur tablette par accident, mono-colonne centrée largeur max 480 px.

#### D17 — Anti-pattern « lecture cours interrompue par paywall »

**Quoi** : La lecture des cours est **gratuite** intégralement. Le paywall n'apparaît **que** lors d'une tentative de Mode 2 / Mode examen / Fiches / Chat ×20.
**Pourquoi** : Les cours sont du contenu de référence — couper l'accès aux notions de base démolirait la confiance et la valeur perçue. La monétisation porte sur la **pratique** (Modes) et le **chronométrage** (examens), pas sur la lecture passive.
**Impact aval** : Aucun écran de lecture cours ne déclenche un paywall. Aligné avec le SPEC CAP-2 (consultation libre) vs CAP-3-4-5-6 (modes et premium).

## 2026-06-04 — Update : Posture émotionnelle + Multisensoriel (motion / audio / haptic)

**Skill** : `bmad-ux` (intent `Update`)
**Mode** : Fast (utilisateur a tranché les 3 paramètres clés en amont)
**Auteur** : Delano Roosvelt

### Trigger

L'utilisateur a signalé après merge de PR 0.3 (AppLogger) que trois dimensions transverses manquaient dans le scope :

1. **Animations** — « tout doit être animé »
2. **Sons + vibrations** — « pense aussi car c'est très important »
3. **Émotion** — « les interfaces ou il y a de l'émotion on doit dégager de l'émotion »

### Décisions

#### D-UX-Update-1 — Posture motion : *micro-interactions partout, transitions sobres*

**Quoi** : Tout élément interactif réagit visuellement au tap ; chaque changement d'état est animé ; les transitions inter-écrans restent standard (`slide page` 200 ms). Cible 60 fps sur Android Go.
**Pourquoi** : Compromis entre l'ambition « tout est animé » et la contrainte non-négociable « téléphones modestes ». Hero animations cinématiques et parallaxes rejetées (perf). Lottie écarté en V1 (taille APK + perf).
**Comment** : Tokens motion ajoutés à DESIGN.md § Animations & motion (`fast` 120 / `standard` 200 / `emphasis` 300 / `celebration` 600 ms + easings standardOut/In/emphasized + stagger 50 ms). Package retenu : `flutter_animate` + Implicit widgets natifs.
**Impact aval** : Story 0.10 (tokens) reçoit AC5 « Motion tokens » ; Story 0.13 (atoms) reçoit anim `tap feedback` 120 ms sur boutons + `selection click` haptic ; Story 0.14 (feedback) inclut désormais les overlays célébration animés.

#### D-UX-Update-2 — Posture audio : *standard 8-12 sons, budget ≤ 500 KB*

**Quoi** : Catalogue 12 sons OGG embarqués (tap, success_soft/strong, error_soft/strong, complete, levelup, badge, payment_ok, notification, streak, chat_send). Total cible ≤ 500 KB. Aucun streaming.
**Pourquoi** : L'utilisateur a tranché « Standard 8-12 sons ≤ 500 KB » plutôt que « minimaliste » ou « pas de son ». Justification : ambiance riche, mais pas de coût data en runtime et budget APK maîtrisé.
**Coupures globales** : silencieux Android → muet ; Mode Examen → muet ; setting Profil « Sons activés » (par défaut ON).
**Impact aval** : Story 0.14 reçoit AC9 « AudioService » + AC10 « overlays célébration » ; assets déclarés en `pubspec.yaml`. Production des sons à planifier en story P0 séparée (placeholders OGG silence si nécessaire pour livrer Story 0.14 avant production audio).

#### D-UX-Update-3 — Posture haptic : *discret mais présent, API Flutter native*

**Quoi** : Catalogue `selection / light / medium / heavy / success(séquence) / error(séquence)` via `HapticFeedback.*`. Pas de dépendance externe pour le haptic.
**Pourquoi** : Renforce chaque confirmation positive/négative significative sans alourdir l'APK. API native = zéro perm supplémentaire au-delà de ce que Flutter déclare déjà.
**Coupures globales** : batterie < 15 % → muet ; Mode Examen → muet ; setting Profil « Vibrations activées » (par défaut ON). Le mode silencieux Android n'affecte PAS le haptic (vibration différente du son).
**Impact aval** : Story 0.14 reçoit AC8 « HapticService » avec gestion des coupures.

#### D-UX-Update-4 — Posture émotionnelle explicite, 7 moments cibles

**Quoi** : Nouvelle section EXPERIENCE.md § Emotional Posture qui mappe 7 moments (onboarding, lecture, bonne/mauvaise réponse, paywall, examen, santé faible) sur une émotion cible, un levier principal et un anti-pattern. Règles d'écriture émotionnelle (« Pas tout à fait » plutôt que « Faux »).
**Pourquoi** : L'utilisateur a explicité que l'émotion est un levier produit (rétention, persévérance, usage), pas un vernis. Les UJs existants (Fatou, James) le supposaient implicitement ; on rend la règle décidable.
**Impact aval** : guide la copy de toutes les stories E1-E6. Pas de nouveau code direct mais référence dans la définition de chaque story de feature ayant un écran utilisateur.

#### D-UX-Update-5 — Choreography par moment clé, source unique

**Quoi** : Table de choreography (anim + son + haptic + copy) pour ~17 moments clés dans EXPERIENCE.md § Multisensoriel. Source unique consultée par les implémentations.
**Pourquoi** : Évite que chaque story réinvente la combinaison ; garantit la cohérence émotionnelle globale.
**Impact aval** : Quand une story implémente un écran, elle consulte cette table pour appliquer la combinaison correcte. Les services `HapticService` et `AudioService` (Story 0.14) sont les points d'appel.

- Flows (Flow 1-8) : inchangés (les flows existants restent valides, la couche multisensorielle s'y greffe via la table de choreography).
- Information Architecture, Accessibility Floor, Responsive & Platform : inchangés.
