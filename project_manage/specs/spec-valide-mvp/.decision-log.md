### Décisions de scoping prises pendant la distillation

#### D1 — Découpage en 6 capabilities alignées sur les 6 phases du Decoupage MVP

**Quoi** : Un `CAP-N` par phase MVP, pour faciliter le mapping `SPEC → PRD → epics → stories`.
**Pourquoi** : Le Decoupage MVP est déjà structuré en phases hebdomadaires avec des livrables utilisateur clairs. Forcer un autre découpage (par feature, par module) aurait introduit une couche de traduction inutile. La table de bord en tête de `phases-mvp.md` formalise la correspondance.
**Impact aval** : Les epics produits par `bmad-create-epics-and-stories` devraient se mapper 1:1 sur les CAP, avec un sprint par phase.
**Réversibilité** : moyenne — on peut redécouper plus tard, mais les IDs CAP-1..6 seront stables (Spec Law rule 6).

#### D2 — `phases-mvp.md` créé comme companion spec-authored plutôt que d'absorber tout dans le kernel

**Quoi** : Les détails opérationnels des 6 phases (livrables, à développer, critères d'acceptation par phase) sont dans le companion `phases-mvp.md`, pas dans le kernel.
**Pourquoi** : Le kernel doit rester lean (Spec Law rule 8). Le détail phase par phase est un catalogue multi-items — exactement le cas d'usage d'un companion (cf. doc de la skill : « multi-item catalogs »).
**Impact aval** : `bmad-create-epics-and-stories` lit le companion pour construire les epics.

#### D3 — `glossary.md` créé comme companion spec-authored

**Quoi** : Termes métier (Mode 1/2/3, santé scolaire, sous-système, etc.) et techniques (idempotence, App Check, AppLogger, etc.) regroupés dans un glossaire dédié.
**Pourquoi** : Les termes apparaissent dans CAP, Constraints, Non-goals — sans glossaire, downstream interprète différemment. Le glossaire est un cas explicite mentionné dans la doc de la skill (« glossary.md »).
**Impact aval** : Toutes les skills downstream peuvent référer au glossaire au lieu de re-définir.

#### D4 — Les 4 archis et 4 docs partagées listées comme **adopted companions**, pas comme sources

**Quoi** : `doc/tech/Valide School App Architecture.md`, `Valide Cloud Function Architecture.md`, `Valide School Package Architecture.md` ET `doc/partage/BASE-DE-DONNEES.md`, `ALGORITHMES.md`, `CONTRATS-API.md`, `DONNEES-REFERENCE.md` → listés en `companions:` (chemin relatif), pas en `sources:`.
**Pourquoi** : Ces docs sont **load-bearing** pour downstream (Winston en archi consulte l'archi mobile/backend ; Bob en story consulte la surface partagée). Elles ne sont pas absorbées dans le SPEC — elles restent owned par leurs originating contexts. C'est exactement la définition d'« adopted companion » dans la doc de la skill.
**Impact aval** : Downstream lit ces fichiers en plus de SPEC.md.

#### D5 — Le Decoupage MVP (markdown + PDF) listé en **sources**, pas en companions

**Quoi** : `doc/metier/Valide Decoupage MVP.md` et `MVP Valide School.pdf` → listés en `sources:`.
**Pourquoi** : Tout le contenu load-bearing de ces documents a été absorbé dans SPEC.md + `phases-mvp.md`. Downstream n'a pas besoin de les lire — ils sont là pour traçabilité narrative (justifications de fond, prose color).
**Impact aval** : Évite à downstream de relire un PDF de 619 KB pour comprendre le scope.

#### D6 — 5 hypothèses formalisées dans `Assumptions`

**Quoi** : Agrégateur Mobile Money exposera un webhook signé exploitable ; volume de contenu suffisant au lancement ; latence `europe-west1` OK depuis le Cameroun ; matrice profil → matières validée fin P1 ; roster de 6 agents BMAD v6.8.0 suffisant.
**Pourquoi** : Ces points ne sont pas confirmés par les sources, mais le SPEC en dépend implicitement. Les formaliser rend le risque visible plutôt que silencieux.
**Impact aval** : Chaque assumption à valider explicitement avant la phase qui en dépend.

#### D7 — 7 questions ouvertes formalisées dans `Open Questions`

**Quoi** : Périmètre exact des séries techniques en V1 ; inclusion série E ; déclencheur du port iOS ; volume minimal de contenu pour la Success signal ; échelle de mention vs barème officiel ; comportement pour élève sans école ; plafond chat strict vs gradiant.
**Pourquoi** : Le Decoupage MVP est précis sur les règles produit mais flou sur ces décisions de scope qui doivent être tranchées avant ou pendant la phase concernée.
**Impact aval** : `bmad-prd` doit trancher ces points en mode `Coaching` ou via `bmad-advanced-elicitation`.

#### D8 — Success signal exprimé en parcours canonique mesurable, pas en métriques business

**Quoi** : La Success signal décrit un test concret (deux élèves réels, deux devices réels, parcours bout en bout, connexion fluctuante) plutôt que des objectifs « X utilisateurs, Y conversion ».
**Pourquoi** : C'est ce que le Decoupage MVP appelle à valider en fin de P6. Les métriques business sont en aval du MVP. La Success signal doit être **testable ou démontrable** (Spec Law rule 5).
**Impact aval** : Le test de fin de P6 est défini ici. La QA pourra dérouler cet exact scénario.

#### D9 — Constraints formulées en termes de « ce que ça interdit / impose »

**Quoi** : Chaque constraint nomme explicitement ce qu'elle écarte (« pas de cache custom », « pas de paiement carte », « pas d'iOS », etc.) plutôt qu'une simple observation.
**Pourquoi** : Spec Law rule 3 : « Constraints actually bend design decisions ». Une constraint qui ne rule out rien est décoration. Toutes les constraints listées écartent au moins une option de design plausible.
**Impact aval** : Winston (architecte) lit ces constraints comme des règles dures à respecter dans `architecture.md`.

#### D10 — 11 Non-goals explicites au lieu d'un minimum

**Quoi** : 11 entrées dans Non-goals, alors que Spec Law rule 4 n'exige qu'« au moins une ».
**Pourquoi** : Le Decoupage MVP est très clair sur ce qui est **dans** le scope ; ce qui est **hors** scope est plus flou et invite à des dérives (« est-ce qu'on pourrait ajouter… »). Lister explicitement les 11 limites prévient ces dérives.
**Impact aval** : Quand un stakeholder demandera une feature, John (PM) répondra en pointant le Non-goal concerné.
